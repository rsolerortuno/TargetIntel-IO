# v0.4.0 feasibility cards and reports

## Canonical surfaces and scope

The inspected canonical Markdown card surface is `targetintel.hypothesis_cards`
(`make_target_card`, `write_target_card`, and `write_top_target_cards`).  The
canonical standalone report surface is `targetintel.html_reports`
(`make_target_html_report`, `write_target_html_report`, and
`write_top_html_reports`).  The operational card script is
`scripts/05_generate_cards.py`; HTML reports are invoked by
`scripts/06_generate_html_reports.py` and the normal pipeline.  Both select
the union of top targets from the three existing rank columns and then sort by
antibody/IO, biomarker, and small-molecule rank.  Current cards contain
identity, stable role and direction, intent scores/ranks, evidence,
confidence/limitations, and a next-experiment section.  Existing fixture and
snapshot coverage remains the authority for deterministic pipeline outputs.

No structured card/report renderer currently exists.  Therefore this issue
does not create one.  `FeasibilityReportSection.to_dict()` is a namespaced,
serializable presentation value for a future existing structured surface; it
does not replace cards or reports.

## Additive architecture

The only supported flow is:

```
existing deterministic ranking -> existing target selection -> existing card/report
construction -> optional feasibility decoration -> final rendering
```

`targetintel.feasibility.presentation` accepts only already-constructed Issue
404 `ModalityFeasibilityAnnotation` values (or their
`ModalityAssessmentWithFeasibility` wrapper).  It validates exact target and
identifier type, annotation/profile identity, version, modality uniqueness,
coverage, and observation-reference uniqueness before rendering.  It does not
fetch Open Targets data, resolve symbols, read caches, build profiles, or
compose modality assessments.  No CLI option or automatic pipeline chaining is
introduced; Issue 406 owns an explicit end-to-end demonstration.

The section is immutable and deterministic.  Its ID includes the version,
target identity/type, canonical modality annotation identities, source release
and verification state, observation references, coverage, contradictions,
limitations, and research/no-score/no-ranking/no-recommendation boundaries.
It excludes timestamps, paths, processes, logs, and filenames.  Modalities are
ordered antibody, small molecule, PROTAC, other clinical; dimensions are
clinical precedence, tractability, doability, safety.

## Presentation semantics

When supplied, the canonical renderers append a clearly-labelled **Target
feasibility — research-only** section.  It preserves modality-specific
observation references separately from target-level contextual references.
Small-molecule observations are never inferred to be PROTAC observations.
Clinical precedence is descriptive only; doability is not a readiness score;
no modality is selected or recommended.

Coverage remains descriptive, not confidence, evidence strength, a percentage,
or a ranking weight.  It retains observed, partial, not available, not
applicable, conflicting, retrieval-failed, and source-level not-observed
semantics through the approved coverage and observation references.  Missing
records do not imply negative feasibility.  Contradictions retain every
reference and are not resolved or averaged.

Safety wording is deliberately constrained: absence of a retrieved safety
signal is not evidence that a target is safe.  Safety statements remain scoped
to successfully retrieved records; unavailable or failed retrieval makes no
safety conclusion.

Values are safely escaped for Markdown and HTML.  The section copies only
canonical observation identities, not raw source payloads, and accepts neither
arbitrary dictionaries, transport payloads, profiles, cache data, hidden
reasoning, nor LLM output.

## Compatibility and isolation

Without feasibility input, both canonical renderers execute their exact legacy
paths and produce unchanged content, filenames, ordering, target selection,
scores, roles, rationale, evidence, and metadata.  With it, only the appended
research-only section is added.  Feasibility cannot alter scores or components,
roles, intent labels/scores/ranks, target ordering/selection, controls,
benchmarks, features, or sensitivity analyses.

Issue 405 does not change Obsidian export, grounded synthesis, v0.3 exports,
README, release material, package version, or dependencies.  This remains a
hypothesis-generation and target-triage presentation aid, not therapeutic,
clinical, safety, or go/no-go advice.
