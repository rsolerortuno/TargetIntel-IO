# Reviewed evidence persistence (v0.3.0)

## Boundary

This contract persists exactly one `EvidenceItem` only after an explicit
`EvidencePersistenceRequest` references an Issue 306 `EvidencePromotionResult`
with status `promoted` and `persisted: false`. Persistence does not promote a
candidate, create review decisions, retrieve documents, call an LLM, validate
citations externally, score or rank targets, render reports, or export data.

The request contains an opaque persistence actor ID distinct from the human
reviewer ID, a logical store identifier, the expected store schema version when
applicable, and deterministic references to the promotion, item, decision,
audit, and candidate. Actor identity is always caller supplied: environment
variables, credentials, paths, and implicit approvals are not inputs.

## Eligibility and provenance

The service accepts only a promotion with one present `EvidenceItem`. It checks
the promotion identity, request-to-promotion references, item identity, and the
manual-review provenance retained by Issue 306. The retained chain includes
request/response provenance through candidate, card, audit, packet, decision,
promotion, item, and receipt IDs. It requires source-document identity and a
source-content hash, but neither receipts nor requests copy the source document.
The established `EvidenceItem` finalization validator is reused unchanged.

## Idempotency and append-only storage

The deterministic idempotency key covers the request format version, promotion
result ID, item ID, review-decision ID, logical store ID, and canonical item
payload hash. Timestamps, paths, process and run identifiers, provider timing,
and logs are excluded. The initialized `EvidenceStore` must be configured with
the same explicit `logical_store_id` as the request; an absent or mismatched
store identity fails closed before lookup or write. This identity is
operator-supplied bookkeeping, never derived from a filesystem path. The
existing `EvidenceStore` public lookup, `insert_finalized_item`, and retrieval
methods are reused; its immutable insert transaction is the local backend's
strongest available atomic operation.

If the item ID is absent, one immutable item is inserted and retrieved again for
canonical round-trip verification. An exact existing representation returns
`already_persisted` without a write. The same item ID with different canonical
content returns `content_conflict` and never overwrites or merges evidence.
There is no automatic retry, migration, redirection, update, deletion,
supersession, or retraction.

## Receipts and security

Receipts are immutable and deterministic apart from their optional operational
timestamp. They retain status, IDs, actor, store/schema identifiers, payload
hash, write flag, sanitized ordered codes, and the explicit statement that
persistence is neither scientific nor clinical validation. Only `persisted` and
`already_persisted` report `persisted: true`.

Stable expected failures use sanitized codes, never tracebacks, SQL, connection
strings, secrets, hidden reasoning, or complete source text. Request parsing
rejects unknown fields, credential fields, and hidden-reasoning fields. The
service accepts an initialized local store object; it performs no network,
subprocess, dynamic import, filesystem discovery, or downstream processing.

An operational CLI, lifecycle management, downstream scoring/ranking/report
consumption, supersession, and retraction remain deferred.
