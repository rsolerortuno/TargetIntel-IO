# v0.3.0 LLM execution CLI

`python -m targetintel.llm` is an operational boundary for provider output. It
does not retrieve literature, construct `EvidenceItem` records, verify
citations, alter scoring, or make therapeutic recommendations.

## Commands

`validate-request REQUEST_JSON` parses an explicit UTF-8 JSON object into an
immutable `LLMRequest` without selecting a provider or writing files.

`execute REQUEST_JSON --provider mock --mock-script SCRIPT --output RUN_DIR`
performs one deterministic offline MockProvider call. Mock scripts are JSON
objects keyed by request ID; each value is a MockOutcome-shaped object.

`execute REQUEST_JSON --provider ollama --model MODEL --output RUN_DIR
--allow-local-network` permits one Ollama call only to a loopback URL. No
network is invoked without the explicit flag. `--dry-run` validates provider
selection and writes a `not_executed` response without generation or transport.
Ollama setup, service startup, downloads, cloud authentication, custom headers,
and non-loopback URLs are out of scope.

`inspect-run RUN_DIR` verifies artifact hashes and reports completion, provider,
and response status. It never executes a provider or changes the directory.
`--show-response` displays only the sanitized response contract.

## Request and artifacts

The request is a top-level object with the fields of `LLMRequest`. Unknown
fields are rejected. Exactly one of `source_text` and
`structured_source_content` is required. Text is preserved verbatim;
structured content is frozen and canonically serialized. Credentials in
contract mappings are rejected recursively.

Runs contain UTF-8 canonical `request.json`, `response.json`, and, after those
succeed, `manifest.json`. The manifest declares format
`targetintel-llm-run-v1`, SHA-256 hashes for request and response artifacts,
completion state, provider, model, and response status. Existing output
directories fail closed unless `--overwrite` is supplied.

Request/response identities use existing canonical contracts and exclude
operational timestamps, durations, and paths. Operational fields are visibly
nested separately. Source documents are never printed in normal command output;
there are no environment sourced prompts, implicit retrieval, dynamic imports,
SDK discovery, subprocesses, retries, fallback providers, or unsafe
deserialization.

## Stable exit codes

| Code | Meaning |
| --- | --- |
| 0 | success (including validated dry-run/not-executed artifact) |
| 2 | CLI usage error |
| 3 | invalid request or mock script |
| 4 | invalid provider configuration |
| 5 | retryable provider failure or timeout |
| 6 | permanent provider failure |
| 7 | malformed provider output |
| 8 | output conflict or write failure |
| 9 | invalid or tampered run artifacts |
| 10 | unexpected internal failure |

All tests use MockProvider or injected Ollama transports; neither Ollama,
network access, model weights, GPUs, nor API keys are required.
