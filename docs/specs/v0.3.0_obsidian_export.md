# Deterministic Obsidian export v0.3.0

Issue 310 is the explicit persistence boundary after Issues 308 (reviewed snapshots) and 309 (grounded target synthesis). It accepts only a successful immutable `GroundedTargetSynthesis` (or the narrowly scoped helper that extracts one from a `generated` result) and an explicit immutable `ObsidianExportRequest`.

There is no default vault, discovery, home-directory scan, environment-derived root, CLI, pipeline chaining, LLM invocation, evidence-store access, retrieval, scoring, ranking, or scientific transformation. The caller supplies a vault root and safe relative `.md` note path. Roots must be existing directories; absolute paths, traversal, NUL bytes, symlink roots/destination components/destinations, and paths resolving outside the root fail closed.

Requests contain versioned schema identity, synthesis identity, path, actor, controlled collision policy, renderer/frontmatter versions, canonical unique tags, and optional operational timestamp. The vault root and timestamp are serialized operationally but excluded from request identity. Secret/credential and hidden-reasoning keys are recursively rejected.

`build_obsidian_export_plan` is pure: it reuses Issue 309's `render_grounded_synthesis_markdown` without changing its scientific text, and wraps it in stable YAML-compatible frontmatter (JSON-quoted scalar subset, no YAML dependency). Frontmatter has canonical fields for synthesis/snapshot/inventory provenance, target/context/purpose/counts, research and non-clinical boundaries, explicit no-score/no-ranking values, and sorted tags. Plans carry complete UTF-8 bytes, SHA-256, byte count, and deterministic identity; operational vault paths do not enter plan identity.

`persist_obsidian_export` checks request/plan linkage and plan integrity, revalidates containment and symlinks, then uses a temporary file in the destination directory, flushes it, and publishes it by hard link. This avoids overwriting a concurrently created differing user note. `fail_if_exists` always returns `collision`; `idempotent_same_content` returns `already_current` only for byte-identical existing content. Differing content is never changed and no backups are made.

Receipts are immutable and deterministic over status, identities, content metadata, write flag, and sanitized codes. Operational absolute path/timestamp remain separate. Statuses are `written`, `already_current`, `invalid_request`, `invalid_synthesis`, `invalid_plan`, `identity_mismatch`, `unsafe_path`, `symlink_escape`, `vault_not_found`, `vault_not_directory`, `collision`, and `write_error`; only `written` represents a new write.

Broader reporting integration, operational CLI, automatic exports, vault configuration, indexing, synchronization, and plugin installation are deferred.
